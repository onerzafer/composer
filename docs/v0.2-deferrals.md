# v0.2 Deferrals

Things we deliberately deferred during v0.1 implementation. Each entry says **what** was deferred, **why** (it was acceptable for v0.1), and **where** the gate currently lives (so v0.2 implementation knows where to start).

This document is the **canonical** list — when you see `TODO(v0.2)` in source, `it.skip` in tests, or "deferred to v0.2" in `tasks.md`, the explanation lives here.

---

## 1. Catalog caching across composes

**Status**: deferred from v0.1.

**What**: Each `compose()` invocation calls `loadCatalog()` → `compileCatalog()`. In v0.1 that runs `tsx`'s transpiler against `catalog/index.ts` from scratch per compose. Cold-start is ~30s on the first compose in a fresh workspace.

**Why deferred**: Single-shot composes (the dominant agent workflow) complete in the SC-001 budget of 60s. Repeated composes in the same process (tests, batch operations) pay the cost more than once.

**Why it's the v0.2 priority**: The skipped test `tests/e2e/agent-loop.test.ts > idempotence on adapter-next` exists to flush this out. Even a 120s timeout doesn't suffice; 2x cold-start dominates.

**Implementation hint**: Cache the `CompiledCatalog` keyed by `(workspaceRoot, catalogIndexPath, mtimes)` in `@composer/core/src/workspace/`. Invalidate when any catalog file mtime changes. Tests will skip-then-pass once it lands.

## 2. Template prep files (`*.prep.ts`)

**Status**: deferred from v0.1.

**What**: Per-primitive prep functions for non-trivial data shaping. Spec FR-010, FR-011, FR-017 reserve them.

**Why deferred**: v0.1's adapter-next demonstrates that pure Handlebars + the `{{slot family variant}}` helper covers the slot-registry use case without prep. The constitution's 30-line discipline (V) is satisfied. Adding the sandboxed-prep loader adds dependency risk + complexity that v0.1 doesn't need.

**Where the gate lives**: `packages/core/src/pipeline/phases/render.ts` — checks for `<name>.prep.ts` and throws `RenderFailedError` with a clear "deferred to v0.2" message if found. The sandbox itself (`runPrepInSandbox`) is already implemented in `packages/core/src/render/sandbox.ts` — only the loader/transpile pipeline is missing.

**Implementation hint**: Use the existing `tsx` loader (same as `loadCatalog`) to import the prep module, then run its default export inside `runPrepInSandbox`. Pass node + `{slots, tokens}` ctx. Already-built primitives — wire in the new pre-render hook.

## 3. Staging-dir + atomic-rename hardening

**Status**: deferred from v0.1.

**What**: v0.1's `commit` phase writes directly to target paths after drift-check passes. v0.2 should write to `.composer/staging/<output-paths>` first, then atomic-rename to target paths only after all writes succeed.

**Why deferred**: For single-file compositions (Page → one page.tsx), direct write + drift-check pre-pass is functionally atomic. Multi-file compositions (when adapter-hono lands with API endpoints + schemas) need the staging pass for true atomicity if one write fails mid-flight.

**Where the gate lives**: `packages/core/src/pipeline/phases/commit.ts` — module comment notes the v0.1 simplification.

**Implementation hint**: Two-pass: (1) write all files to `.composer/staging/<path>`; (2) rename each staged file to its target. Failure during step 1 → cleanup staging. Failure during step 2 → keep already-renamed files (impossible to roll back) but report explicitly.

## 4. Per-line source maps

**Status**: deferred from v0.1.

**What**: Current source map (`.composer/cache/sourcemap.json`) records one entry per primitive node, spanning the whole emitted block. `composer explain <file>:<line>` works at block granularity. v0.2 should record per-line ranges so explain points at the exact JSON field that produced any specific line.

**Why deferred**: Block-level explain is sufficient for the v0.1 debugging story ("which spec node produced this?"). Per-line requires the renderer to track output position per template variable, which is bigger surgery.

**Where the gate lives**: `packages/core/src/pipeline/phases/render.ts` — `buildSourceMapEntries` returns one entry per file with `line_start: 1, line_end: <last>`. Replace with per-block tracking from a Handlebars AST walk.

## 5. Catalog hot-reload

**Status**: resolved (landed alongside deferral #1's content-hash cache).

**What**: When a human edits `catalog/primitives/hero.ts` while an MCP server is running, the server now picks up the edit on the very next `discover`/`scaffold` call — no restart needed.

**How it landed**: `loadCatalog()` (`packages/typescript/src/loader.ts`) re-hashes the catalog's source files (`hashCatalogSources`) on every call — hashing is not itself cached, only the parsed module/compiled catalog keyed by the resulting hash is. `discover()`/`scaffold()` (`packages/core/src/api/discover.ts`, `scaffold.ts`) already call `loadCatalog` fresh on every invocation rather than holding a workspace-lifetime reference, and the MCP server (`packages/mcp/src/server.ts`) has no request-level caching of its own — `callTool` re-invokes the tool handler, which re-invokes `discover`/`scaffold`, on every call. So a cache hit (unchanged catalog) skips the ~30s `tsx` transpile, while any edit changes the hash and forces a fresh load — no explicit file watcher needed. A no-op fs.watch/chokidar layer would only save the ~0.1ms hashing cost per call, which isn't worth the added dependency/complexity.

**Verified by**: `tests/integration/mock-agent.test.ts` — "MCP server picks up a catalog edit between two discover calls, no restart (v0.2 deferral #5)" drives a real `createServer()` instance through `discover` → edit `catalog/index.ts` on disk → `discover` again, and asserts the second call's primitive list reflects the edit.

## 6. MCP HTTP transport

**Status**: deferred from v0.1 (research R5).

**What**: MCP server currently exposes stdio only. v0.2 could add HTTP transport for remote-agent / web-IDE scenarios.

**Why deferred**: All v0.1 target agents (Claude Code, Codex, Gemini CLI, OpenCode, Antigravity) support stdio. Local-only by definition; no network exposure surface.

**Where the gate lives**: `packages/mcp/src/server.ts` — `start()` wires `StdioServerTransport`. Add an alternative `startHttp({ port })` method when needed.

## 7. Brownfield ingestion (`composer ingest`)

**Status**: deferred from v0.1 (spec §15 sketch).

**What**: CLI commands to derive primitives from existing source (`composer ingest react src/components/`, `composer ingest openapi spec.yaml`, etc.). Outputs to `design/catalog/ingested/` quarantine for human review + promotion.

**Why deferred**: Brand-new feature surface. v0.1 supports adoption via `composer init --bare` (build from scratch) or `composer init --extends @composer/adapter-next` (use the reference adapter).

**Where the gate lives**:
- `packages/typescript/src/loader.ts` already ignores `<workspace>/catalog/ingested/` structurally (FR-023 G1 fix).
- `tasks.md` CLI namespace reserves `composer ingest` and `composer promote`.
- Each ingester plugin (`@composer/ingest-react`, `@composer/ingest-openapi`, etc.) is its own package.

## 8. Migration codemods (`composer migrate`)

**Status**: deferred from v0.1 (non-goals + spec §4).

**What**: When a primitive's contract changes (e.g., a required field added), specs using it potentially break. `composer migrate --from 1.0 --to 1.1` should run codemods to update spec JSON in place.

**Why deferred**: Composer hasn't shipped a v1 yet, so there are no version bumps to migrate. Defer until at least one major catalog version shift is in flight.

**Where the gate lives**: `tasks.md` CLI namespace reserves the command.

## 11. LSP server

**Status**: cut from v0.1 entirely.

**What**: A Language Server speaking the LSP protocol so human catalog authors get completion + diagnostics + go-to-spec in their IDE.

**Why cut**: Per the user redirect during brainstorming — the MCP server already covers every LSP-equivalent for *coding agents* (schemas via scaffold, diagnostics via validate, hover docs via fieldGuidance, go-to-spec via `composer explain`). Vanilla TypeScript LSP covers most of what *human catalog authors* need.

**Where the gate lives**: nowhere — no scaffolding for LSP exists. Revisit only if human catalog authoring becomes a real pain point.

## 12. Constrained-decoding integration

**Status**: deferred from v0.1 (research R5 + non-goals).

**What**: Wire Composer's catalog JSON Schema into agent generation paths via Outlines, XGrammar, llguidance, or native Anthropic structured outputs — making invalid JSON unrepresentable during agent generation.

**Why deferred**: Important per README §12 ("from post-hoc Zod validation to decode-time grammar enforcement"). Not on the critical path for "agent can attach and compose."

**Where the gate lives**: nowhere — `scaffold()` already returns JSON Schema (via `zod-to-json-schema`); v1 can wire that into per-agent constrained-decoding setup as a follow-up.

---

## 13. Daemon mode (`composerd`)

**Status**: deferred from v0.1 (research R5 + design §A.4 in source design doc).

**What**: Persistent background daemon that caches catalogs, holds compiled validators, watches files. MCP/CLI talk to the daemon over IPC.

**Why deferred**: v0.1's one-shot model works for solo agent attach. Daemon is the asymptote — only worth building once multiple surfaces (MCP + CLI + LSP + future tooling) demand fast hot-path access to a shared compiled state.

**Where the gate lives**: nowhere — the engine is currently process-local. Daemon would be `@composer/daemon`.

---

## 14. BNF/EBNF formal grammar export

**Status**: deferred from v0.1 (non-goals).

**What**: Auto-emit a formal grammar specification (BNF/EBNF) from the catalog's Zod schemas so external tooling can consume the contract without reading TS source.

**Why deferred**: Nice-to-have for ecosystem interoperability. No concrete consumer demands it yet. Should be cheap to produce once needed (Zod schemas → JSON Schema via `zod-to-json-schema` → BNF derivation).
