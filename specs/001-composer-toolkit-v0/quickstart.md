# Quickstart — Composer v0.1

> Adopt Composer in a Next.js project, compose your first feature with an LLM agent, and inspect the result. ~5 minutes.

## Prerequisites

- Node.js 20 LTS or newer
- An existing Next.js project (App Router) — or run `npx create-next-app` first
- A Composer-capable agent: Claude Code, Codex CLI, or any other MCP-supporting agent

---

## 1. Install and initialize (≈30s)

From your Next.js project root:

```bash
npx @composer/cli init --extends @composer/adapter-next
```

You should see:

```text
✓ wrote composer.json
✓ installed @composer/adapter-next@1
✓ scaffolded design/ from adapter
✓ wrote .gitignore entries
✓ ran sample compose() — src/app/page.tsx was generated

Next:
  - Read design/guidelines.md to understand the catalog
  - Attach your agent (see "Connecting an agent" below)
  - Try: composer compose home
```

What just happened:

| File / dir | Origin | What it is |
|---|---|---|
| `composer.json` | written by init | the project pointer: `{ workspace, engine, extends }` |
| `design/catalog/primitives/*.ts` | copied from adapter | Hero, Section, Card, Form, etc. — TS/Zod schemas with `intent` + `whenToUse` + `whenNotToUse` + examples |
| `design/templates/*.tsx.hbs` | copied from adapter | one Handlebars template per primitive |
| `design/specs/home.json` | written by sample compose | the canonical home page composition |
| `src/app/page.tsx` | generated from spec | banner + source-map comments at the top |
| `.composer/cache/`, `.composer/logs/`, `.composer/staging/` | added to .gitignore | the engine's local working area |

## 2. Inspect the generated file

Open `src/app/page.tsx`. You'll see a banner and per-block source-map comments:

```tsx
// =====================================================================
// DO NOT EDIT. Generated by Composer.
//   spec:    design/specs/home.json
//   inspect: composer explain src/app/page.tsx:LINE
// =====================================================================

// from: design/specs/home.json:7 (Page)
export default function HomePage() {
  return (
    <>
      {/* from: design/specs/home.json:12 (Hero, id=home-hero) */}
      <CenteredHero title="Welcome" ... />
      ...
    </>
  );
}
```

Edits go to the spec (or to the template, or to the catalog). Never to this file.

## 3. Connect an agent

### Claude Code

```bash
npm install -g @composer/skill-claude    # or add to your project's CLAUDE.md skills
```

The skill registers `@composer/mcp` as a Claude Code MCP server. Open your project in Claude Code; the agent will recognize `composer.json` and follow the `discover → scaffold → compose` workflow per the skill prose.

### Any other MCP agent (Codex, Gemini CLI, OpenCode)

Add to your agent's MCP config:

```json
{
  "composer": {
    "command": "npx",
    "args": ["-y", "@composer/mcp"]
  }
}
```

No skill pack needed in v0.1 for non-Claude agents; the MCP tool descriptions guide the agent.

---

## 4. Ask the agent to build something

In your agent's chat:

> Build a pricing page with three plans (Starter, Pro, Enterprise) and a call-to-action.

The agent will:

1. Call `composer.discover()` — learns the catalog, sees `design/specs/home.json` as an example.
2. Call `composer.scaffold({kind: "primitive", primitive: "Page", intent: "pricing page with three plans"})` — gets the Page primitive's schema, skeleton, examples.
3. Compose the JSON in its own context.
4. (Optional) `composer.validate("pricing", json)` — checks structurally + semantically, returns the `would_write` diff.
5. `composer.compose("pricing", json)` — atomically validates, persists `design/specs/pricing.json`, emits `src/app/pricing/page.tsx`.

You'll see new files appear under `src/app/pricing/`.

---

## 5. Edit the spec yourself (the right way)

Open `design/specs/pricing.json` and tweak a value (change a plan name, reorder cards, edit a CTA label).

Then:

```bash
composer compose pricing
```

The generated files update. Idempotent: if you re-run with no changes, nothing happens.

---

## 6. Try editing the generated file (the wrong way)

Edit `src/app/pricing/page.tsx` directly. Change a string value.

Now run compose again:

```bash
composer compose pricing
```

You get:

```text
✗ DRIFT DETECTED
  src/app/pricing/page.tsx has been hand-edited.
  diff:
    - title="Pricing built for teams"
    + title="Pricing for teams"
  
  Options:
    (a) git checkout src/app/pricing/page.tsx && composer compose pricing
    (b) Lift the change into design/specs/pricing.json,
        then composer compose pricing

Exit code: 4
```

Nothing was written. The spec is unchanged. Your hand-edit is intact (in the working tree).

This is the constitution's drift-detection mandate (principle VI). Generated code is an inspectable artifact, not a place you edit.

## 7. Debug generated code

If you hit a runtime error at `src/app/pricing/page.tsx:42`, ask Composer what produced that line:

```bash
composer explain src/app/pricing/page.tsx:42
```

Output:

```text
src/app/pricing/page.tsx:42
  spec:      design/specs/pricing.json:18
  primitive: PricingCard
  node_id:   plan-pro
```

The error is in the data shape of `plan-pro` (edit spec), the rendering of PricingCard (edit template), or the schema (edit catalog). Three places, three layers — never the generated file.

To go the other way:

```bash
composer trace pricing:18
```

```text
design/specs/pricing.json:18 (primitive=PricingCard, node_id=plan-pro)
  → src/app/pricing/page.tsx:42-58
```

---

## 8. Health check

Before pushing to CI:

```bash
composer doctor
```

Reports: primitive sprawl (count + last-used), template+prep LOC discipline (30-line rule), drift state across all generated files, bijection check, stale lockfile, naming hygiene. Exits non-zero on errors; warnings allowed unless `--strict`.

## What you got

In ~5 minutes:

- A Composer-instrumented project
- The agent loop working end-to-end (discover, scaffold, compose)
- Drift detection preventing silent overwrites
- Source-map traversal in both directions
- A clean separation: humans edit catalog + templates + specs; the engine writes source files
- No code written by the LLM. Real source files, type-checked, lintable, debuggable.
