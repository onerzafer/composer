# Feature Specification: Composer works in CommonJS host projects

**Feature Branch**: `002-fix-commonjs-host`

**Created**: 2026-05-28

**Status**: Draft

**Input**: User description: "Fix Composer so it works out-of-the-box inside CommonJS host projects (e.g. a default NestJS app). The tsx loader transpiles the workspace `output.map.ts` to CommonJS and Node's interop double-wraps the default export, so `byPrimitive` resolves to undefined and compose crashes with `Cannot read properties of undefined (reading '<Primitive>')`. The engine's module loaders must unwrap a nested default; `composer init` should also emit a workspace-local `package.json` with `{\"type\":\"module\"}`. Includes a regression test composing inside a CommonJS host fixture and documentation for backend/CommonJS adoption."

**Discovered**: 2026-05-28 while testing the CLI against a stock NestJS (CommonJS) project — a Composer-generated GraphQL API with simulated-DB and simulated-JWT primitives. The full agent loop, atomic compose, drift store, `doctor`, and `explain`/`trace` all worked, and the generated app built, booted, and enforced the simulated JWT — but only after a manual workaround (a workspace-local `package.json` with `{"type":"module"}`) because `composer init`/`compose` first crashed in the CommonJS host.

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Adopt Composer in a CommonJS host project (Priority: P1)

A developer adds Composer to an existing backend project whose `package.json` does **not** declare `"type": "module"` (the default for NestJS, Express, and most CommonJS Node services). They run `composer init` (or hand-author a workspace) and then compose a spec. Today this fails immediately; after this fix it succeeds with no manual intervention.

**Why this priority**: This is the whole feature. Composer claims to emit "hand-quality source code in any target language," but in practice it only works when the host repo happens to be ESM. The majority of existing Node backends are CommonJS, so the engine silently excludes its largest adoption surface. Without this, "works in any project" is false.

**Independent Test**: In a fresh directory whose `package.json` has no `"type": "module"`, run `composer init --bare`; the auto-run sample compose MUST succeed and write its output file, with no `Cannot read properties of undefined` crash from the output-map loader.

**Acceptance Scenarios**:

1. **Given** a host project with `package.json` that has no `"type": "module"`, **When** the developer runs `composer init --bare`, **Then** the workspace is created and the sample compose writes its output file successfully (exit 0).
2. **Given** the same CommonJS host with a valid workspace and spec, **When** the developer runs `composer compose <spec_id>`, **Then** the engine loads `output.map.ts` correctly (its `byPrimitive` map resolves) and emits the expected file(s).
3. **Given** a workspace whose adapter defines an audit module, **When** compose runs in a CommonJS host, **Then** the audit module's default export loads and runs without an interop error.
4. **Given** `composer init` is run in any host, **When** it creates the workspace, **Then** it writes a workspace-local `package.json` containing `{"type":"module"}` so authored workspace modules are unambiguously ESM regardless of the host's module system.

---

### User Story 2 — Backend adopter has the guidance to ship a running app (Priority: P2)

A developer following the docs to instrument a CommonJS backend (e.g. a NestJS GraphQL service) can find the conventions needed for the generated code to build and run — without rediscovering them by trial and error.

**Why this priority**: The engine fix (US1) makes compose work, but a backend adopter still needs to know two host-integration facts to get a *running* app: the generated workspace must not be swept into the host's TypeScript build, and (for NestJS 11 + GraphQL on Express 5) the Apollo Express-5 integration package is required. Documenting these turns "compose succeeds" into "app runs."

**Independent Test**: A developer follows the documentation to instrument a stock CommonJS backend and reaches a building, booting app without consulting the source or this spec.

**Acceptance Scenarios**:

1. **Given** the adoption docs, **When** a developer instruments a CommonJS backend, **Then** the docs state that the workspace directory must be excluded from the host's TypeScript build (the workspace authoring files are tooling, not app source).
2. **Given** the adoption docs, **When** a developer builds a NestJS 11 GraphQL app generated by Composer, **Then** the docs note the `@as-integrations/express5` dependency requirement.

---

### Edge Cases

- **Host is ESM (`"type": "module"`)**: behaviour MUST be unchanged — the existing single-`default` export still resolves; the unwrap logic only descends when the expected shape is one level deeper.
- **Module exports a function directly (audit) vs. an object (output map)**: the unwrap MUST detect the expected shape (a callable for audit, a `byPrimitive` object for the output map) rather than blindly descending, so a legitimately object-shaped `default` is never mis-unwrapped.
- **Workspace already contains a `package.json`**: `composer init` MUST NOT clobber an existing workspace `package.json`; only create it when absent (and, if present without `"type":"module"`, surface a clear note rather than silently failing later).
- **Host `package.json` later flips to/from `"type":"module"`**: compose MUST remain correct in both states with no re-init required.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The engine MUST load the workspace output map correctly when the resolved module is a CommonJS-transpiled default export that is wrapped one level deeper than expected (i.e. unwrap a nested `default` when the outer object lacks the expected `byPrimitive` shape).
- **FR-002**: The engine MUST apply the same nested-`default` tolerance to the audit module loader, resolving to the audit function whether it is exported as `default`, a nested `default`, or a named `audit` export.
- **FR-003**: The unwrap behaviour MUST be shape-aware and MUST NOT alter the result for ESM hosts where the default export already has the expected shape (no regression for existing ESM projects).
- **FR-004**: `composer init` MUST create a workspace-local `package.json` declaring `{"type":"module"}` (for both `--bare` and `--extends` modes), and MUST NOT overwrite an existing workspace `package.json`.
- **FR-005**: The project MUST include an automated regression test that performs a compose inside a CommonJS host fixture (a fixture whose `package.json` omits `"type":"module"`) and asserts the output file is written.
- **FR-006**: Documentation MUST describe CommonJS / backend adoption: the workspace `{"type":"module"}` convention, excluding the workspace directory from the host's TypeScript build, and the `@as-integrations/express5` requirement for NestJS 11 GraphQL hosts.

### Key Entities

- **Workspace module**: a TypeScript file authored in the workspace (`catalog/index.ts`, `output.map.ts`, audit) that the engine loads at compose time via the TypeScript runtime loader. Its on-the-wire export shape depends on the host project's module system.
- **Host project**: the directory that contains `composer.json`; its `package.json` `type` field determines whether the loader sees ESM or CommonJS-transpiled modules.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: `composer init --bare` followed by the auto-run sample compose succeeds (exit 0, output file written) in a host project whose `package.json` has no `"type": "module"` — verified in an automated test.
- **SC-002**: 100% of existing tests continue to pass (no regression for ESM hosts) after the change.
- **SC-003**: A developer can take a stock CommonJS backend and reach a composed, building, booting app using only the documentation — demonstrated by the NestJS GraphQL example (simulated-DB + simulated-JWT) that serves an authenticated query and rejects an unauthenticated one.

## Assumptions

- The root cause is the `tsx`→CommonJS interop double-wrapping the default export; the fix is in the engine's module loaders plus an `init` convention, not in `tsx` itself.
- The workspace directory is logically separate from the host application's source and may carry its own `package.json` without affecting the host build.
- Verified manually on 2026-05-28 against NestJS 11 (`@nestjs/graphql` 13, `@apollo/server` 5, `@as-integrations/express5`); the regression test will encode the minimal reproduction (bare workspace in a CommonJS host), not the full NestJS stack.
- This is a v0.1.1 hardening fix layered on the `001-composer-toolkit-v0` engine; it depends on that code being present (branch `002-fix-commonjs-host` is cut from `001`).
